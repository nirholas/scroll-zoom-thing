# Automating Parallax Assets with AI Agents

This guide documents how AI agents — specifically Claude Code — integrate into the CSS 3D Parallax Scrolling workflow for this project. It covers what agents can reliably automate, what they cannot, how the four built-in skills work, prompt patterns, multi-agent pipelines, and practical cost considerations drawn from real usage.

---

## What Agents Can and Cannot Automate

The honest answer is that agents are good at the mechanical, repetitive, and structural parts of this workflow and unreliable at the aesthetic ones.

**Agents reliably automate:**

- Generating Stable Diffusion or Midjourney prompts for individual depth layers given a scene description
- Writing the Python scripts that batch-convert PNG/EXR/WebP source images to AVIF
- Scaffolding the initial `mkdocs.yml`, `home.html`, and `home.css` from a written brief
- Writing documentation pages (like this one) from the code and comments
- Running `mkdocs build`, reporting errors, and applying targeted fixes
- Adjusting CSS numeric values (depth, perspective, translateZ, scale) within defined bounds
- Cross-checking that all AVIF files referenced in `home.html` actually exist in `docs/assets/hero/`

**Agents require human judgment for:**

- Deciding whether a generated image looks right — agents cannot see rendered output without vision tooling
- Choosing the right depth order for a specific scene (what goes in front vs. behind)
- Color grading and mood: whether the sky layer should be warmer or cooler
- Final parallax speed ratios: the relationship between layers is a feel decision, not a math problem
- Typography and layout choices on the homepage

Understanding this boundary matters because it determines how you structure the agent loop. You do not ask an agent to "make the parallax look good." You ask it to "generate four depth-ordered prompts for a mountain scene, then convert the output PNGs to AVIF, then wire them into `home.html`." You review the result visually and give specific feedback.

## The Four Built-In Claude Code Skills

Claude Code skills are stored in `skills/` and invoked with a slash command prefix. Each skill is a Markdown file containing a system prompt and structured instructions that Claude follows when you invoke it.

### `/setup-parallax`

Scaffolds the full project structure. When invoked on a fresh MkDocs installation, it:

1. Creates `docs/overrides/home.html` with the perspective/layer skeleton
2. Creates `docs/assets/stylesheets/home.css` with the CSS 3D transform rules
3. Adds `custom_dir: docs/overrides` to `mkdocs.yml` if not present
4. Adds `extra_css: [assets/stylesheets/home.css]` to `mkdocs.yml`
5. Creates `docs/assets/hero/` if it does not exist
6. Writes placeholder `<img>` tags in `home.html` for each expected layer

Invoke it from the project root:

```bash
claude
# Then type:
/setup-parallax
```

Or pass arguments for the number of layers:

```
/setup-parallax layers=5
```

### `/generate-prompts`

Takes a scene description and returns four Stable Diffusion (or Midjourney) prompts, one per depth layer, ordered from furthest-back to foreground:

```
/generate-prompts "foggy mountain valley at dawn, pine forest in midground, mossy rocks in foreground"
```

Expected output format:

```
Layer 1 (sky/far background):
  Prompt: misty mountain peaks at dawn, soft pink sky, volumetric fog, no foreground elements,
          ultra-wide aspect ratio, photorealistic, –ar 16:9

Layer 2 (mid-distance):
  Prompt: pine forest silhouette, fog rolling through trees, no sky visible, dark green tones,
          depth of field, photorealistic, –ar 16:9

Layer 3 (near midground):
  Prompt: rocky mountain path, scattered boulders, low ground fog, pine needles on ground,
          sharp detail, photorealistic, –ar 16:9

Layer 4 (foreground):
  Prompt: close-up mossy rocks and ferns, sharp foreground, blurred background, morning dew,
          photorealistic, –ar 16:9
```

The skill enforces that each prompt explicitly excludes elements belonging to other depth layers. This prevents the AI image generator from placing sky in the foreground layer or rocks in the sky layer — a common failure mode when prompts are written without depth awareness.

### `/convert-images`

Writes and executes a Python script that batch-converts source images (PNG, WebP, JPEG, or EXR) in a given directory to AVIF using `cavif` or `pillow-avif-plugin`. The agent:

1. Scans the source directory for image files
2. Determines whether `cavif` (preferred, better quality) or `pillow` (fallback) is available
3. Writes a Python script with configurable quality and effort settings
4. Runs the script
5. Reports the output sizes and conversion ratios

```
/convert-images source=./raw-layers/ output=docs/assets/hero/ quality=80 effort=6
```

The generated script (not the skill file itself — the script the skill writes) looks like:

```python
#!/usr/bin/env python3
"""Batch convert layer images to AVIF. Generated by /convert-images skill."""
import subprocess
import sys
from pathlib import Path

SOURCE = Path("./raw-layers/")
OUTPUT = Path("docs/assets/hero/")
QUALITY = 80
EFFORT = 6

OUTPUT.mkdir(parents=True, exist_ok=True)

for src in sorted(SOURCE.glob("*.png")):
    dest = OUTPUT / src.with_suffix(".avif").name
    result = subprocess.run(
        ["cavif", "--quality", str(QUALITY), "--effort", str(EFFORT),
         "-o", str(dest), str(src)],
        capture_output=True, text=True
    )
    if result.returncode != 0:
        print(f"FAILED: {src.name} — {result.stderr.strip()}", file=sys.stderr)
    else:
        ratio = dest.stat().st_size / src.stat().st_size
        print(f"OK: {src.name} → {dest.name} ({ratio:.1%} of original)")
```

### `/tune-layers`

Reads the current CSS transform values from `home.css` and the layer positions from `home.html`, then proposes adjusted values to change the depth feel. The tuning is based on the mathematical relationship between `perspective`, `translateZ`, and the scale factor needed to maintain apparent size:

```
scale = perspective / (perspective - translateZ)
```

Invoke it with a description of what feels wrong:

```
/tune-layers "the foreground moves too fast relative to the sky layer"
```

The agent reads `home.css`, identifies the `translateZ` values for each layer, calculates the current speed ratios, and outputs a diff of proposed changes. It does not apply the changes automatically — it prints the CSS with proposed values highlighted so you can review before accepting.

## Claude Code CLI Integration

Claude Code is invoked from the project root. The `claude` binary reads the `CLAUDE.md` file in the current directory automatically:

```bash
cd /path/to/scroll-zoom-thing
claude
```

For one-shot tasks without an interactive session:

```bash
claude -p "Convert all PNGs in ./raw-layers to AVIF in docs/assets/hero/ at quality 80"
```

For file access, Claude Code reads and writes files in the working directory tree. No special configuration is needed — if the shell user has read/write access to a file, so does the agent.

For shell tool access (running `mkdocs build`, `cavif`, `python`), the project's `.claude/settings.json` controls which commands are allowed without an explicit permission prompt:

```json
{
  "permissions": {
    "allow": [
      "Bash(mkdocs build*)",
      "Bash(mkdocs serve*)",
      "Bash(cavif *)",
      "Bash(python *)",
      "Bash(pip install *)",
      "Bash(ls *)",
      "Bash(find docs/ *)"
    ]
  }
}
```

Keep the allow list narrow. There is no reason to give the agent `Bash(rm *)` or `Bash(git push *)`.

## The CLAUDE.md File: Behavioral Guidelines

The `CLAUDE.md` at the project root sets four behavioral guidelines that shape every agent interaction in this repo:

**Think Before Coding:** The agent is instructed to surface assumptions and ask clarifying questions before writing code. For the parallax workflow this means: if you say "add a new layer," the agent will ask whether you want it in front of or behind the current foreground, what image file to use, and what `translateZ` value to start with — rather than guessing.

**Simplicity First:** The agent will not add "flexibility" that was not requested. If you ask it to convert images, it writes a straight conversion script — not a configurable class hierarchy with YAML-driven settings.

**Surgical Changes:** When editing `home.css`, the agent touches only the property values you asked about and leaves adjacent rules alone. It matches the existing formatting (tabs vs. spaces, property ordering) rather than reformatting the file.

**Goal-Driven Execution:** For multi-step tasks the agent states a verification plan before executing:

```
1. Add layer-5 div to home.html → verify: grep "layer-5" docs/overrides/home.html returns a match
2. Add .layer-5 CSS rule to home.css → verify: mkdocs build exits 0
3. Check layer-5.avif exists in docs/assets/hero/ → verify: ls docs/assets/hero/layer-5.avif
```

These guidelines matter because without them, LLMs tend to over-engineer solutions, make wide-ranging edits that break unrelated code, and proceed confidently when they should ask.

## The AGENTS.md File

Where `CLAUDE.md` sets behavioral guidelines, an `AGENTS.md` file (placed at the project root or in `agents/`) provides agent-specific workflow context: the directory layout, which commands are safe to run, which files are safe to modify, and the expected output format for each skill.

A minimal `AGENTS.md` for this project:

```markdown
# AGENTS.md

## Safe to read: all files
## Safe to write: docs/overrides/, docs/assets/, mkdocs.yml
## Never write: src/, contracts/, public/

## Commands available without prompting:
- mkdocs build
- mkdocs serve --dev-addr=127.0.0.1:8000
- cavif (if installed)
- python (any .py in this repo)

## Layer naming convention:
- layer-1.avif = furthest back (sky)
- layer-4.avif = closest foreground

## Expected AVIF dimensions: 2560x1440 or 1920x1080
## Expected AVIF quality: 75-85 (higher = better, larger)
```

The key difference from `CLAUDE.md`: `AGENTS.md` describes the project's structure and conventions. `CLAUDE.md` describes how the agent should reason and behave. Both are read by Claude Code on startup.

## Pattern 1: Scene Description to Four Layer Prompts

This is the starting point for every new parallax hero. The agent takes a plain English scene description and produces four image-generation prompts, one per CSS layer.

**System prompt structure (inside the `/generate-prompts` skill):**

```
You are an expert at decomposing a scene description into depth-separated image layers
suitable for CSS 3D parallax. Each layer must be a standalone, photorealistic image
that looks correct when composited with the others. Each layer must contain ONLY
elements that belong at its depth. Include no other elements. Use PNG transparency
for foreground layers.
```

**User prompt structure:**

```
Scene: "rainy city street at night, neon signs reflecting on wet pavement, pedestrians
with umbrellas, taxis in foreground"

Generate four prompts:
- Layer 1 (furthest back, sky/buildings): ...
- Layer 2 (mid-distance, street level): ...
- Layer 3 (near midground, reflections/umbrellas): ...
- Layer 4 (foreground, taxis/drops): ...

Rules:
- No layer should contain elements from another layer's depth.
- Foreground layers (3, 4) must be generated with transparent background.
- Each prompt must specify aspect ratio 16:9.
- Each prompt must end with: photorealistic, octane render, –q 2
```

**Expected output format** — the agent returns a fenced code block containing the four prompts, labeled and ready to paste into an image generation tool.

## Pattern 2: Image Processing Pipeline

After the images are generated (externally, by a human using Midjourney or Stable Diffusion), the agent handles the mechanical conversion step.

```bash
claude -p "I have PNGs in ./raw-layers/. Convert them to AVIF at quality 82, effort 7,
and place them in docs/assets/hero/ named layer-1.avif through layer-4.avif
in alphabetical order of the source filenames."
```

The agent:
1. Lists `./raw-layers/` to see what files exist
2. Writes a Python script to do the conversion
3. Runs it with the Bash tool
4. Confirms each output file exists and prints its size
5. Reports any files that failed conversion

The agent does not assume `cavif` is installed. It checks `which cavif` first and falls back to `pillow-avif-plugin` if needed, installing it with `pip install pillow pillow-avif-plugin` before proceeding.

## Pattern 3: Full Site Scaffold from Description

This is the highest-leverage agent task: going from a written brief to a working MkDocs site with a parallax hero.

```bash
claude -p "$(cat <<'EOF'
Scaffold a MkDocs Material site for a product called 'Axiom'. The hero should have
4 AVIF layers already placed in docs/assets/hero/. Use indigo as the primary color,
slate dark scheme by default. The homepage should have no visible MkDocs content
below the hero — full-screen parallax only. Include a nav with: Home, Docs, API.
Create: mkdocs.yml, docs/index.md, docs/overrides/home.html, docs/assets/stylesheets/home.css.
Run mkdocs build at the end and report any errors.
EOF
)"
```

The agent creates all four files, then runs `mkdocs build --strict`. If the build fails (which it often does on the first pass due to a missing nav page), it reads the error, fixes the specific issue, and rebuilds.

This pattern typically takes 2-3 iterations to reach a clean build. The agent is reliable at fixing structural errors (missing files, invalid YAML, broken Jinja2 blocks). It is not reliable at pixel-perfect layout — expect to tune the CSS manually after the scaffold is in place.

## Pattern 4: Iterative Tuning Loop

The tuning loop is a human-in-the-loop pattern. The agent reads current values, proposes a change, you review it in the browser, and give feedback.

```
You: /tune-layers "sky layer barely moves, foreground moves too much"

Agent: Current values:
  .layer-1 { transform: translateZ(-400px) scale(1.4); }  /* sky */
  .layer-2 { transform: translateZ(-200px) scale(1.2); }  /* midground */
  .layer-3 { transform: translateZ(-100px) scale(1.1); }  /* near mid */
  .layer-4 { transform: translateZ(0px) scale(1.0); }     /* foreground */

  With perspective: 1000px, the scroll speed ratios are approximately:
  layer-1: 0.29x, layer-2: 0.56x, layer-3: 0.77x, layer-4: 1.0x

  Proposed fix: increase layer-1 translateZ from -400 to -200 to make sky move more,
  decrease layer-4 translateZ from 0 to -50 to slow foreground slightly.

  Proposed values:
  .layer-1 { transform: translateZ(-200px) scale(1.25); }
  .layer-4 { transform: translateZ(-50px) scale(1.05); }

  Apply? (yes/no)

You: yes

Agent: [applies changes to home.css, runs mkdocs build, confirms success]
```

The math behind speed ratios: in CSS 3D perspective, a layer at `translateZ(-d)` moves at `(perspective - d) / perspective` of the scroll speed relative to a layer at `translateZ(0)`. The agent knows this formula and uses it to reason about the proposed changes before suggesting them.

## Pattern 5: Documentation Generation

The agent can write documentation pages (like this one) from the codebase. The prompt structure:

```
You are writing technical documentation for a MkDocs Material site. Read the following
files and write a complete guide covering [topic list]. The guide should be at least
2000 words, use professional technical writing, and include code blocks for all
configuration examples.

Files to read:
- CLAUDE.md
- docs/overrides/home.html
- docs/assets/stylesheets/home.css
- mkdocs.yml
- skills/ (all files)
```

The agent reads the files, extracts the relevant details, and writes the documentation. This is the most reliable documentation workflow: the agent cannot hallucinate file contents it has actually read.

## Giving Agents Shell Tool Access

The commands this project's agents need:

```bash
mkdocs build          # build the site, check for errors
mkdocs serve          # local preview
cavif                 # AVIF conversion (optional, preferred)
python                # run conversion scripts
pip install           # install missing Python packages
ffmpeg                # video to AVIF (if animated layers are needed)
ls, find, cat         # file inspection
```

Safety considerations:

- Never grant `git push` without human review. The agent making a change to `home.css` should not be able to push it to production unreviewed.
- Never grant `rm -rf` or any recursive delete. The agent should write new files, not delete existing ones.
- Scope `pip install` to the local virtualenv (`pip install --user` or inside an activated venv) to avoid polluting system Python.
- If using Claude Code in a CI environment, run it in a container with no network access beyond what is strictly needed.

## Agent Evaluation Loop

The practical loop for any agent task in this workflow:

```
1. Write: agent generates or modifies files
2. Build: agent runs mkdocs build --strict
3. Review: human opens http://127.0.0.1:8000 and inspects result
4. Feedback: human describes what is wrong
5. Iterate: return to step 1
```

Steps 1 and 2 are fully automated. Steps 3 and 4 always require a human. The agent cannot evaluate visual output without vision tooling (a screenshot + multimodal model), and even with vision tooling the feedback loop is slow and error-prone for subtle CSS issues.

Expect 3-5 iterations on a new scaffold and 1-2 iterations on a targeted change like "move this layer forward" or "make the text larger."

## Using Ollama for Fast Depth/Position Tuning

Ollama runs LLMs locally. For the numerical tuning loop (adjusting `translateZ`, `scale`, `perspective`), a fast local model is preferable to a cloud model because the round-trip latency matters when you're making many small adjustments.

Setup:

```bash
ollama pull mistral:7b
# or for better reasoning:
ollama pull qwen2.5-coder:14b
```

The model needs to understand the CSS 3D perspective formula. Inject it as a system message:

```
You are a CSS 3D parallax tuning assistant. The key formula is:
  apparent_scroll_speed = (perspective - translateZ) / perspective
where perspective is the value set on the parent container (typically 800px-1200px).
A layer at translateZ(-400px) with perspective 1000px moves at 60% of native scroll speed.
Given current layer positions and the user's description of what feels wrong, propose
specific translateZ and scale values. Respond with only the changed CSS properties.
```

For a 14B model running locally on Apple Silicon or a mid-range GPU, response latency is under 5 seconds — fast enough for an interactive tuning session.

## Multi-Agent Workflows

Complex tasks can be split across specialized agents running in sequence:

**Stage 1: Prompt generation agent**

```bash
claude -p "Generate Midjourney prompts for a 4-layer parallax hero: futuristic city skyline.
Output to prompts/city-hero.txt"
```

**Stage 2: (Human) Run prompts through Midjourney, save output PNGs to ./raw-layers/**

**Stage 3: Image processing agent**

```bash
claude -p "Convert all PNGs in ./raw-layers/ to AVIF in docs/assets/hero/ at quality 80"
```

**Stage 4: Site wiring agent**

```bash
claude -p "Update docs/overrides/home.html to reference the AVIF files in docs/assets/hero/.
Verify mkdocs build passes."
```

**Stage 5: Documentation agent**

```bash
claude -p "Read docs/overrides/home.html and docs/assets/stylesheets/home.css.
Write docs/guide/layers.md explaining how the parallax layers are structured."
```

This staged approach is more reliable than a single "do everything" prompt because each stage has a clear, verifiable output and human review points are natural between stages.

## The `.claude/commands/` Directory

`.claude/commands/` is the local project-level equivalent of skills. Files placed here are available as slash commands in any Claude Code session started from this project:

```
.claude/
  commands/
    rebuild.md       → /rebuild
    lint-css.md      → /lint-css
    check-layers.md  → /check-layers
```

The difference from `skills/` is scope. Skills in `skills/` are the canonical, documented skills for this project — they are checked into version control and documented. Commands in `.claude/commands/` are workspace-local shortcuts, often more task-specific or experimental.

A useful `/check-layers` command:

```markdown
# check-layers

Check that every AVIF file referenced in docs/overrides/home.html
exists in docs/assets/hero/.

Steps:
1. Read docs/overrides/home.html and extract all .avif src attributes
2. For each extracted path, check if the file exists with `ls`
3. Report any missing files
4. Report any files in docs/assets/hero/ that are NOT referenced in home.html
```

## Testing Agent Output

Before accepting agent-generated files, verify:

**Valid YAML (`mkdocs.yml`):**

```bash
python -c "import yaml; yaml.safe_load(open('mkdocs.yml'))" && echo "OK"
```

**Valid Jinja2 (templates in `docs/overrides/`):**

```bash
mkdocs build 2>&1 | grep -i "jinja\|template\|error"
```

**Valid CSS:**

```bash
npx stylelint "docs/assets/stylesheets/*.css"
```

**Full build:**

```bash
mkdocs build --strict
```

The strict build is the integration test. It catches broken links, missing files, invalid YAML, and Jinja2 errors in a single command.

## Common Agent Mistakes

### Wrong File Paths

Agents frequently confuse `docs_dir` relative paths with project root relative paths. A stylesheet at `docs/assets/stylesheets/home.css` is referenced in `mkdocs.yml` as `assets/stylesheets/home.css` — the `docs/` prefix is omitted. Agents often write the full path, breaking the build.

Fix: In your prompt, specify paths explicitly: "reference it in mkdocs.yml as `assets/stylesheets/home.css` (relative to docs_dir, not project root)."

### Case Sensitivity on Linux

On macOS (case-insensitive filesystem), `Layer-1.avif` and `layer-1.avif` are the same file. On Linux (case-sensitive), they are not. Agents generating HTML that references `Layer-1.avif` while the actual file is `layer-1.avif` will produce a site that works locally on a Mac and breaks in CI on Linux.

Fix: Always use lowercase filenames for all assets. Enforce this in `AGENTS.md`: "All asset filenames must be lowercase."

### Missing `{{ super() }}` in Jinja2 Blocks

An agent overriding the `{% block styles %}` block and omitting `{{ super() }}` will silently strip all of Material's built-in CSS, breaking the entire theme. The page will render as unstyled HTML.

Fix: In your agent prompt or `AGENTS.md`, include the rule: "When overriding a Jinja2 block, always call `{{ super() }}` unless you explicitly intend to replace the entire block's output."

### Indented Front Matter

YAML front matter must start at column 1 with `---`. An agent that generates a Markdown file with indented front matter will produce a page that MkDocs cannot parse:

```markdown
  ---          ← WRONG: indented
  template: home.html
  ---
```

Fix: The CLAUDE.md "Simplicity First" guideline helps here — the agent will not add indentation for unclear reasons.

## Cost Considerations

Not all tasks justify using a cloud model. A rough guide:

| Task | Recommended model |
|---|---|
| Prompt generation (4 layer prompts) | Cloud (Claude Sonnet, GPT-4o) — creative, benefits from RLHF |
| Image conversion script | Local (Qwen 7B, Mistral 7B) — mechanical, well-defined |
| CSS tuning (numerical values) | Local (Qwen 14B) — arithmetic, short context |
| Full site scaffold | Cloud (Claude Sonnet) — architectural decisions matter |
| Documentation generation | Cloud (Claude Sonnet) — long output, quality matters |
| Check-layers audit | Local (any model) — trivial file inspection |

For a typical new parallax site, the cloud model usage (scaffold + prompts + docs) costs on the order of $0.05-0.20 in API credits. The image generation cost (Midjourney, Stable Diffusion) is orders of magnitude higher and is the actual cost driver for this workflow.

## The PAI Docs as a Production Example

`docs.pai.direct` is a production documentation site built with this exact stack: MkDocs Material, CSS 3D parallax hero, AVIF layers, and agent-assisted maintenance.

The PAI team uses the agent workflow as follows:

- When a new product version ships, an agent reads the changelog and generates a draft of the "What's New" page. A human editor reviews and publishes.
- When AVIF layers need updating (seasonal or campaign-driven), the `/generate-prompts` skill produces new image prompts. The team runs them through their image generation pipeline externally, then uses `/convert-images` to process the outputs.
- Documentation pages are drafted by the `/generate-docs` agent (a custom `.claude/commands/` command specific to their setup) and edited by the technical writer.
- `mkdocs build --strict` runs in GitHub Actions on every PR. The PAI team does not merge PRs with failing builds.

The key insight from the PAI workflow: agents are most valuable as first-draft generators, not as final editors. Every agent-generated page goes through human review before it is published. The agents handle the structural and mechanical work; humans handle the judgment calls. This division of labor — which is exactly what `CLAUDE.md` enforces through its "Think Before Coding" and "Simplicity First" guidelines — is what makes the workflow sustainable at scale.
